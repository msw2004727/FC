## 重要問題

### 1. 非管理者申請後，可能看不到自己的 pending 申請，通知也可能不送出

相關位置：

- `js/modules/tournament/tournament-friendly-state.js:74-76`
- `js/firebase-crud.js:214-231`
- `firestore.rules:941-947`
- `js/modules/tournament/tournament-friendly-notify.js:153-163`

問題說明：

前端 `_loadFriendlyTournamentDetailState()` 會直接呼叫 `listTournamentApplications(tournamentId)`，也就是讀整個 `applications` collection。但 Firestore rules 的 application read 只允許：

- 賽事管理者讀取
- 或該 application 的 team 成員讀取單筆

Firestore query 不是 client-side filter；如果一般申請者嘗試 list 全部 applications，會因為其中可能含有其他隊伍的申請而被 rules 擋掉。程式目前 catch 後 fallback 成 `fallbackApplications`，通常是空陣列。

可能影響：

- 使用者送出申請後，畫面可能仍顯示可報名，因為 state 沒讀到自己的 pending application。
- 通知 wrapper 依賴 afterState 找「新建立的 application」，若 list 被擋，可能找不到新申請，導致主辦者/委託人沒有收到通知。
- 同一隊可能重複送申請，直到 deterministic doc id merge 才緩解，但 UX 仍會混亂。

修正建議：

- 管理者才允許 list 全部 applications。
- 一般使用者改成只讀自己可負責隊伍的 deterministic doc，例如 `applications/ta_${teamId}`。
- 或改用 rules 可驗證的 constrained query，但需確認 Firestore rules 能接受該 query。
- rules 端補強 application doc id 與 `teamId` 對齊，避免任意 doc id。

建議驗收：

- 一般 captain/leader 送出申請後，刷新仍看到「審核中」。
- 另一隊伍的申請不可被該使用者讀取。
- 主辦者/creator/delegate 能收到申請通知。

### 2. 審核通過流程不是原子操作，可能出現 entry 已建立但 application 還是 pending

相關位置：

- `js/modules/tournament/tournament-friendly-detail.js:186-208`
- `js/firebase-crud.js:236-273`
- `js/modules/tournament/tournament-friendly-state.js:45-49`

問題說明：

approve 流程目前是分段寫入：

1. 檢查名額
2. `upsertTournamentEntry()`
3. `updateTournamentApplication()`
4. `_persistFriendlyTournamentCompatState()` 同步 `registeredTeams`

如果第 2 步成功但第 3 或第 4 步失敗，就會留下半套狀態。

可能影響：

- entries 已有該隊伍，但 application 仍是 pending。
- 管理者刷新後可能同時看到「已核准隊伍」與「待審申請」。
- `registeredTeams` summary 可能不同步，列表與詳情數字不一致。

修正建議：

- 新增單一 `reviewTournamentApplication()` 寫入流程，用 Firestore batch 或 transaction 一次處理：
  - 更新 application status/review meta
  - 建立或更新 entries/{teamId}
  - 更新 tournaments/{id}.registeredTeams summary
- quota check 也應放入 transaction 內，避免兩個管理者同時 approve 超過名額。

建議驗收：

- 模擬 application update 失敗時，不應留下 entry。
- 兩個管理者同時 approve 最後一個名額時，只能成功一筆。
- application、entry、registeredTeams 三者狀態一致。

### 3. 建立賽事時 root 與 host entry 不是原子操作，host entry 失敗會產生半壞賽事

相關位置：

- `js/modules/tournament/tournament-manage.js:328-333`
- `js/modules/tournament/tournament-friendly-state.js:92-100`
- `firestore.rules:960-965`

問題說明：

建立賽事時先 `createTournamentAwait(createData)`，再寫 `entries/{hostTeamId}`。第二步如果失敗，目前只 `console.warn`，不阻止建立完成。

detail state 會在讀取時補一個虛擬 host entry，但 roster self-join rules 需要實際存在 `entries/{teamId}`，所以畫面看似正常，底層資料可能不完整。

可能影響：

- 使用者看到主辦隊已在參賽名單，但 members subcollection 寫入被 rules 擋掉。
- 後續名單加入、統計、同步都可能異常。

修正建議：

- 建立賽事 root 與 host entry 改成 batch 寫入。
- 若暫時不能 batch，host entry 寫入失敗時應 rollback root 或明確顯示建立失敗。
- 不建議吞錯後繼續顯示成功。

建議驗收：

- 模擬 host entry 寫入失敗時，不能留下 root-only tournament。
- 建立後 Firestore 一定存在 `tournaments/{id}/entries/{hostTeamId}`。

### 4. guest / deep link allowGuest 被 roster wrapper 吃掉

相關位置：

- `js/modules/tournament/tournament-friendly-roster.js:299-309`
- `js/modules/tournament/tournament-friendly-detail.js:29-36`
- `js/modules/tournament/tournament-detail.js:8-9`
- `app.js:1810`

問題說明：

`tournament-friendly-roster.js` 包裝 `showTournamentDetail(id)` 時沒有接第二個參數 `options`，因此外部呼叫 `showTournamentDetail(id, { allowGuest: true })` 會被轉成只傳 `id`。

可能影響：

- deep link 或訪客允許檢視賽事詳情時，仍可能被 `_requireLogin()` 擋掉。
- 與 event/team detail 的 guest deep link 行為不一致。

修正建議：

```javascript
async showTournamentDetail(id, options) {
  await _tournamentFriendlyRosterLegacy.showTournamentDetail.call(this, id, options);
  // 後續原本 hydrate roster 的邏輯不變
}
```

建議驗收：

- 未登入狀態從 LIFF/deep link 開賽事詳情，若 caller 傳 `allowGuest: true`，應可進入公開詳情。
- 一般 app 內點擊仍維持需要登入的流程。

## 中型問題

### 1. 隊伍 leader 可看到建立按鈕，但儲存會被 `admin.tournaments.entry/create` 前置閘擋住

相關位置：

- `js/modules/tournament/tournament-core.js:254-273`
- `js/modules/tournament/tournament-manage.js:237-240`
- `js/modules/tournament/tournament-helpers.js:103-108`
- `firestore.rules:882-887`

問題說明：

建立按鈕是否顯示是看 `_canCreateFriendlyTournament()`，只要使用者是某隊 captain/leader 就可以看到。但 `handleCreateTournament()` 開頭又額外要求：

```javascript
admin.tournaments.create OR admin.tournaments.entry
```

如果某位 team leader 是一般 user，且沒有 `admin.tournaments.entry`，就會發生「可以開 modal，但按儲存顯示權限不足」。Firestore rules 其實允許 hostTeam captain/leader 建立。

修正建議：

- 前端建立守衛統一為：
  - `admin.tournaments.manage_all`
  - 或 `_canCreateFriendlyTournament(currentUser)`
- 不要讓 `admin.tournaments.entry` 成為建立賽事的必要條件。
- 補測試：leader 無 entry permission 仍可建立自己隊伍主辦的賽事。

### 2. applications rules 缺少資料形狀限制

相關位置：

- `firestore.rules:941-949`
- `js/modules/tournament/tournament-friendly-detail.js:129-139`

問題說明：

目前 application create 只檢查：

- tournament exists
- 已登入
- 有 teamId
- 是該 team captain/leader

但沒有強制：

- 初始 `status` 必須是 `pending`
- `requestedByUid` 必須等於 `request.auth.uid`
- doc id 必須與 `teamId` 對齊
- 可寫欄位白名單
- `teamId` 不可在 update 時被改掉

可能影響：

- captain/leader 可直接建立非 pending 狀態的 application。
- 可偽造 requestedByUid/requestedByName。
- 可用多個 doc id 建立重複申請或髒資料。

修正建議：

- rules 新增 `isValidTournamentApplicationCreate()`。
- 建議 doc id 採固定規則，例如 `ta_${teamId}` 或直接使用 `teamId`。
- create 僅允許固定欄位，且 `status == 'pending'`、`requestedByUid == request.auth.uid`。
- update 僅允許 manager 修改 review 欄位與 status，並鎖住 teamId/requestedByUid。

### 3. root summary 同步失敗被吞掉，可能造成列表與詳情不一致

相關位置：

- `js/modules/tournament/tournament-friendly-state.js:29-63`
- `js/modules/tournament/tournament-render.js:171-175`
- `js/modules/tournament/tournament-detail.js:321-329`

問題說明：

`_persistFriendlyTournamentCompatState()` 同步 `registeredTeams` 失敗時只 `console.warn`，但列表、舊 detail tab、統計仍有地方依賴 `registeredTeams`。

可能影響：

- 子集合 entries 是正確的，但列表顯示報名數錯誤。
- 舊 tab 或 fallback 畫面顯示隊伍不完整。

修正建議：

- 最好併入審核 transaction。
- 若仍保留獨立 sync，失敗應可重試或顯示可觀測錯誤。
- 長期可考慮讓列表 summary 欄位改成 `approvedTeamCount` / `pendingTeamCount`，並由同一個原子流程維護。

### 4. 舊資料若缺 `mode/creatorUid/hostTeamId`，非 admin 更新可能被 immutable rules 擋住

相關位置：

- `js/modules/tournament/tournament-core.js:184-225`
- `js/api-service.js:216-221`
- `js/firebase-crud.js:1207-1221`
- `firestore.rules:889-938`

問題說明：

前端更新時會透過 `_buildFriendlyTournamentRecord()` 正規化資料，補上 `mode`、`creatorUid`、`hostTeamId` 等欄位。Rules 對非 admin 更新要求 immutable 欄位完全等於 resource 原值。

如果舊 tournament 缺少這些欄位，前端補上的預設值可能被視為修改 immutable 欄位，導致 creator/delegate/host manager 更新失敗。admin 會繞過此檢查。

修正建議：

- 優先做資料 migration/backfill，確保所有 tournament root 都有 `mode`、`creatorUid`、`hostTeamId`。
- 或 rules 加過渡邏輯，明確允許 legacy 缺欄位補正，但只能補成安全值。
- 加一個 regression test：legacy 缺 `mode` 時，非 admin record manager 的正常 edit 行為是否符合產品預期。

### 5. 結束賽事 UI 與 handler 權限不一致

相關位置：

- `js/modules/tournament/tournament-manage.js:80-88`
- `js/modules/tournament/tournament-manage.js:370-380`

問題說明：

後台列表的「結束賽事」按鈕只對 admin 顯示，但 handler 實際允許：

- `admin.tournaments.end`
- 或 `_canManageTournamentRecord(t)`

也就是 creator/delegate/host manager 可能具備 handler 權限，但 UI 不顯示。

修正建議：

- 先決定產品規則：
  - 若 record manager 可結束賽事，UI 顯示按鈕。
  - 若只有 admin 可結束賽事，handler 與 rules 都應收緊。
- 不要讓 UI 與 handler 有不同的授權模型。

## 輕型問題

### 1. `reviewFriendlyTournamentApplication()` 沒白名單檢查 action

相關位置：

- `js/modules/tournament/tournament-friendly-detail.js:153-180`

問題說明：

目前只有 `action === 'approve'` 走 approve，其餘非 `reject` 的未知 action 最後會落到 rejected 狀態。

修正建議：

```javascript
const normalizedAction = String(action || '').trim().toLowerCase();
if (!['approve', 'reject'].includes(normalizedAction)) {
  this.showToast('未知的審核操作');
  return;
}
```

並統一後續使用 normalizedAction。

測試內的 `_isTournamentLeaderForTeam()` / `_isTournamentCaptainForTeam()` 仍有 displayName fallback，但正式 helper 只用 UID。這種複製測試容易測到舊邏輯，而不是 production 行為。

### 4. 賽事列表排序規則與「近期賽事」語意不一致

相關位置：

- `js/modules/tournament/tournament-render.js:114`
- `js/modules/tournament/tournament-render.js:171-175`

問題說明：

目前列表以名稱排序。如果頁面語意是「近期賽事」或使用者期待時間排序，名稱排序會讓賽事順序看起來跳。

修正建議：

- 明確定義排序：
  - active：報名中優先，接著 regEnd 或最近 matchDate。
  - ended：最近結束日期或 updatedAt desc。
- UI 若保留名稱排序，需把文案改成比較中性的「全部賽事」。

## 已核實不是問題或風險較低

### members self-join 並非任意登入者可加入

先前需要警惕 `entries/{teamId}/members/{memberUid}` 是否允許任何登入者加入任意隊伍。實際 rules 已有：

- `memberUid == request.auth.uid`
- `isCurrentUserInTeam(teamId)`
- `tournamentEntryExists(teamId)`

所以目前不是「任何登入者可塞名單」的漏洞。

相關位置：

- `firestore.rules:960-977`

後續仍可補強資料形狀，例如 `request.resource.data.uid == memberUid`、欄位白名單、`joinedAt` 型別，但不是最高優先。

## 建議修復順序

1. 先處理 application 讀取模型與通知失效問題。
2. 將審核 approve/reject 改成 transaction/batch。
3. 將建立 root + host entry 改成 batch。
4. 修復 `showTournamentDetail(id, options)` wrapper 遺失 options。
5. 統一建立賽事與結束賽事的 UI/handler/rules 權限模型。
6. 補 rules shape validation 與 regression tests。
7. 最後更新 README 與測試 import 方式。
