# 資料異動監看指引（`change watch`）

## 目的

這份文件提供管理者查看「可疑資料異動監看（`change watch`）」的操作方式與判斷原則。

目前這套機制的定位是：

- 先觀察，不阻擋
- 優先知道「誰在什麼時間改了什麼敏感資料」
- 降低推廣期大改正式流程帶來的風險

這不是權限封鎖機制，也不是完整防護牆。它是一套後端監看與留痕機制。

## 為什麼要監看

專案目前仍有部分正式版流程由前端直接寫入資料庫（Firestore）。

這代表如果有人：

- 用瀏覽器開發者工具（`DevTools`）改 payload
- 不走正常按鈕流程，直接呼叫前端 Firebase SDK
- 刻意繞過 UI 限制去改敏感欄位

就可能留下不合理資料。

因此目前先做的不是全面改成後端處理，而是先補上「可疑異動監看」，讓管理者能追查：

- 哪個帳號改的
- 改了哪個集合
- 改了哪些敏感欄位
- 改前與改後是什麼
- 這次異動屬於哪一種可疑原因

## 系統定位

很重要的一點：

- 後端監看器會看到所有被監看集合的異動事件
- 但只有「符合可疑條件」的事件才會寫入監看日誌
- 一般正常操作通常不會落地成監看紀錄

所以這套系統不是記所有操作，而是記「高信心可疑操作」。

## 實作項目

### 後端函式

目前已上線 4 個 Firestore 異動監看函式：

- 使用者資料監看（`watchUsersChanges`）
- 活動資料監看（`watchEventsChanges`）
- 報名資料監看（`watchRegistrationsChanges`）
- 簽到資料監看（`watchAttendanceChanges`）

### 監看資料區

目前第一批監看 4 個集合：

- 使用者資料（`users`）
- 活動資料（`events`）
- 報名資料（`registrations`）
- 簽到資料（`attendanceRecords`）

### 日誌寫入位置

監看日誌會寫到：

- 異動監看主集合（`changeWatchByDay`）
- 異動監看子集合（`changeWatchEntries`）

完整路徑格式：

- `changeWatchByDay/{dayKey}/changeWatchEntries/{eventId}`

其中：

- `dayKey` 格式為台北時區的 `YYYYMMDD`
- `eventId` 是後端事件 ID，用來避免同一事件重複寫入

### 保留時間

目前監看日誌有自動到期清除（TTL）：

- 到期欄位：`expiresAt`
- 集合群組：`changeWatchEntries`
- 保留天數：約 180 天

## 監看日誌欄位說明

每筆監看日誌主要欄位如下：

| 中文名稱 | 英文代碼 | 說明 |
|---|---|---|
| 日期鍵值 | `dayKey` | 台北時區日期，例如 `20260310` |
| 時間鍵值 | `timeKey` | 台北時區時間，例如 `19:42:15` |
| 集合名稱 | `collectionName` | 被修改的集合，例如 `users` |
| 文件路徑 | `documentPath` | 完整文件路徑 |
| 文件 ID | `documentId` | 被改動的文件 ID |
| 異動類型 | `changeType` | `create`、`update`、`delete` |
| 修改者類型 | `actorType` | 例如 `user`、`system`、`service_account`、`unknown` |
| 修改者 ID | `actorId` | 通常是使用者 UID 或系統身分 |
| 修改者角色 | `actorRole` | 後端推斷出的角色，例如 `user`、`admin` |
| 風險等級 | `riskLevel` | 目前以 `high`、`medium` 為主 |
| 可疑原因 | `reasonCodes` | 這次被記錄的原因代碼 |
| 變更欄位 | `changedFields` | 被判定有異動的敏感欄位 |
| 變更前 | `before` | 敏感欄位修改前值 |
| 變更後 | `after` | 敏感欄位修改後值 |
| 建立時間 | `createdAt` | 後端寫入監看日誌時間 |
| 到期時間 | `expiresAt` | TTL 清除依據 |

## 可監看項目

### 使用者資料（`users`）

重點監看：

- 角色欄位（`role`）
- 手動角色欄位（`manualRole`）
- 權限宣告欄位（`claims`）
- 積分欄位（`exp`）
- 等級欄位（`level`）
- 球隊欄位（`teamId`、`teamName`、`teamIds`、`teamNames`）
- 身分欄位（`uid`、`lineUserId`）

通常不記錄：

- 顯示名稱（`displayName`）
- 頭像（`photoURL`、`pictureUrl`）
- 電話（`phone`）
- 生日（`birthday`）
- 地區（`region`）
- 收藏（`favorites`）
- 社群連結（`socialLinks`）
- 同行者（`companions`）
- 最後登入（`lastLogin`）

### 活動資料（`events`）

重點監看：

- 活動狀態（`status`）
- 已報名人數（`current`）
- 候補人數（`waitlist`）
- 參加者名單（`participants`）
- 候補名單（`waitlistNames`）
- 名額欄位（`max`）
- 主控欄位（`ownerUid`、`creatorUid`、`captainUid`）

### 報名資料（`registrations`）

重點監看：

- 活動 ID（`eventId`）
- 使用者 ID（`userId`）
- 身分欄位（`uid`）
- 報名狀態（`status`）
- 遞補順序（`promotionOrder`）
- 參加者類型（`participantType`）
- 同行者欄位（`companionId`、`companionName`）

### 簽到資料（`attendanceRecords`）

重點監看：

- 活動 ID（`eventId`）
- 使用者 ID（`uid`）
- 狀態欄位（`status`）
- 簽退時間（`checkOutTime`）
- 移除時間（`removedAt`）
- 移除者（`removedByUid`）

## 觸發條件

### 使用者資料（`users`）的可疑條件

會記錄的典型情況：

- 新建立使用者資料時，直接帶入不合理角色、權限、積分或等級
- 一般使用者修改角色、權限、積分、等級
- 一般使用者改動自己的身分欄位（`uid`、`lineUserId`）
- 非信任角色把自己或別人加入新的球隊欄位

通常不記錄的情況：

- 使用者改自己的合理個資
- 登入時正常更新顯示名稱、頭像、最後登入
- 使用者把自己的球隊欄位縮減或清空
- 教練以上角色替他人做球隊歸屬調整

### 活動資料（`events`）的可疑條件

會記錄的典型情況：

- 非活動主控修改活動主控欄位
- 非信任角色修改名額欄位
- 非信任角色修改活動狀態、人數、候補、參加名單，而且變化看起來不像正常報名流程

通常不記錄的情況：

- 活動主控或高權限角色的正常管理操作
- 後端判斷為合理範圍內的報名名單變動

### 報名資料（`registrations`）的可疑條件

會記錄的典型情況：

- 非本人建立別人的報名資料
- 非信任角色修改活動 ID、使用者 ID、同行者欄位、遞補順序
- 非信任角色直接改報名狀態

通常不記錄的情況：

- 本人取消自己的報名
- 活動主控或高權限角色做預期內管理

### 簽到資料（`attendanceRecords`）的可疑條件

會記錄的典型情況：

- 非活動主控替他人新增簽到資料
- 非信任角色替別人改簽到或簽退狀態
- 非信任角色做移除類異動
- 改動簽到身分欄位（活動或使用者）

通常不記錄的情況：

- 活動主控的正常簽到管理
- 高權限角色或系統寫入

## 可疑原因代碼（`reasonCodes`）說明

### 使用者相關

- `user_role_changed`
  - 角色或手動角色被改動
- `user_privilege_field_changed`
  - 權限、積分、等級等敏感欄位被改動
- `user_team_membership_changed`
  - 球隊歸屬欄位被改動
- `user_identity_mismatch`
  - 身分欄位與操作者不一致

### 活動相關

- `event_owner_field_changed`
  - 活動主控欄位被改動
- `event_capacity_changed`
  - 名額欄位被改動
- `event_signup_state_changed`
  - 報名狀態、人數、候補或名單變化看起來不合理

### 報名相關

- `registration_identity_changed`
  - 報名身分欄位或活動綁定欄位被改動
- `registration_status_changed`
  - 報名狀態被直接改動

### 簽到相關

- `attendance_status_changed`
  - 簽到、簽退或相關狀態被改動
- `attendance_removal_changed`
  - 移除類欄位被改動

### 共用

- `delete_sensitive_doc`
  - 敏感文件被刪除

## 管理者監看流程教學

### 1. 打開監看日誌

目前沒有專用後台頁面，管理者請直接到 Firebase Console 查看 Firestore。

查看路徑：

1. 打開 Firestore Data
2. 進入異動監看主集合（`changeWatchByDay`）
3. 找今天的日期文件，例如 `20260310`
4. 進入異動監看子集合（`changeWatchEntries`）

也可以用 collection group 方式直接查：

- `changeWatchEntries`

### 2. 先看風險等級

優先順序建議：

1. `high`
2. `medium`

建議先處理高風險，再看中風險。

### 3. 看修改者

先看這三個欄位：

- 修改者類型（`actorType`）
- 修改者 ID（`actorId`）
- 修改者角色（`actorRole`）

初步判斷方式：

- `system` / `service_account`
  - 多半是系統或後端寫入
- `user` 且角色是 `user`
  - 需要提高警覺
- `admin` / `coach` / `captain`
  - 先結合實際作業流程再判斷

### 4. 看是哪個集合

再看：

- 集合名稱（`collectionName`）
- 文件路徑（`documentPath`）
- 異動類型（`changeType`）

判斷優先順序建議：

1. 使用者資料（`users`）
2. 活動資料（`events`）
3. 報名資料（`registrations`）
4. 簽到資料（`attendanceRecords`）

### 5. 看可疑原因

再看：

- 可疑原因（`reasonCodes`）
- 變更欄位（`changedFields`）

如果同一筆同時有多個原因，優先以最嚴重的原因處理。

### 6. 比對前後值

最後看：

- 變更前（`before`）
- 變更後（`after`）

這一步是最重要的實質判斷。

不要只看原因代碼，要確認：

- 欄位到底改了什麼
- 是否真的超出合理範圍
- 是否造成權限、公平性或資料可信度問題

## 管理者判斷原則

### 優先視為高風險的情況

以下建議直接視為高風險：

- 一般使用者改自己的角色、手動角色、權限、積分、等級
- 一般使用者改自己的 `uid` 或 `lineUserId`
- 非活動主控改活動主控欄位
- 非信任角色大幅改活動名單、人數、狀態
- 非信任角色改報名狀態或遞補順序
- 非活動主控替別人新增或修改簽到紀錄
- 敏感文件刪除

### 可先觀察的情況

以下可以先觀察，不一定立刻處理：

- 教練、領隊、管理者的正常管理操作
- 名單小幅變動但看起來像正常報名流程
- 使用者縮減自己的球隊欄位
- 本人取消自己的報名

### 常見判斷問題

如果看到這種情況，先這樣判斷：

- `actorType` 是 `system`
  - 先確認是不是後端流程，不要立刻當成攻擊
- `actorRole` 是 `captain` 或 `coach`
  - 先確認是不是該活動或球隊的實際管理者
- `reasonCodes` 只有一個，但 `before/after` 很敏感
  - 以實際欄位變化為主，不以原因數量判斷
- 看起來像正常操作但被記錄
  - 列為可能誤報，持續觀察同帳號是否反覆出現

## 建議處理流程

### 高風險事件

建議流程：

1. 先截圖或保留日誌
2. 查對應使用者、活動、報名或簽到資料
3. 確認是否造成權限提升或公平性影響
4. 必要時手動修正資料
5. 視情況限制帳號或暫停該帳號操作
6. 若反覆發生，再回頭做規則或後端封鎖修補

### 中風險事件

建議流程：

1. 先記錄帳號與事件時間
2. 比對同帳號近期是否多次出現
3. 看是否實際造成資料污染
4. 若持續發生，再提升為高風險處理

## 建議管理節奏

推廣期建議：

- 每天至少看一次當天的 `changeWatchByDay/{dayKey}`
- 若有活動日、比賽日、推廣活動，當天提高查看頻率
- 一旦出現高風險事件，當天就完成人工確認

## 限制與注意事項

- 這套系統是事後監看，不是事前阻擋
- 少數正常操作可能被誤記
- 少數模仿正常流程的異常操作可能漏記
- 如果資料是由後端系統寫入，日誌看到的操作者可能是系統身分，不一定是最終使用者
- TTL 會自動清理舊日誌，但不是保證在到期瞬間立刻刪除

## 一句話總結

管理者查看監看日誌時，請依序看：

1. 風險等級（`riskLevel`）
2. 修改者（`actorType`、`actorId`、`actorRole`）
3. 集合與文件（`collectionName`、`documentPath`）
4. 可疑原因（`reasonCodes`）
5. 前後差異（`before`、`after`）

真正的判斷核心不是「有沒有被記錄」，而是「這個異動是否超出這個人本來合理可做的範圍」。
