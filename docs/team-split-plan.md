# 球衣顏色分隊功能計畫書

## 計畫起因

ToosterX 作為運動活動報名平台，核心場景是揪團打球（足球、籃球、排球等）。這類活動到場後的第一件事就是**分隊穿背心**，但目前系統只管「報名」，不管「分隊」，導致：

1. **現場耗時**：主辦到場後才用口頭或 Excel 分隊，20 人活動常花 5-10 分鐘
2. **人數失衡**：沒有事先規劃，常出現一邊多人一邊少人
3. **朋友分散**：純隨機分隊無法滿足「想跟朋友同隊」的需求
4. **重複工作**：每次活動都要重新分，無法沿用上次分組

在活動報名流程中內建分隊機制，讓主辦在活動前就完成分隊，用戶到場直接看手機就知道自己穿什麼顏色，是提升整體使用體驗的自然延伸。

---

## 設計原則

| 原則 | 說明 |
|------|------|
| **色彩即語言** | 全程用色票/色塊溝通，減少文字閱讀成本 |
| **預設聰明** | 2 隊、紅藍、均分 — 主辦 3 秒完成設定 |
| **模式可疊加** | 三種分隊方式不互斥，主辦隨時可手動微調 |
| **最小侵入** | 折疊區塊 + 既有欄位擴充，不改動報名核心流程 |
| **同行者友善** | 預設同隊，降低操作步驟 |

---

## 三種分隊模式

三種模式的定位是「報名時的預設行為」，而非硬限制。不管選哪種模式，主辦在管理端都能手動調整。

| 模式 | 代碼 | 報名時用戶端行為 | 適用場景 |
|------|------|-----------------|----------|
| **用戶自選** | `self-select` | 報名時看到色票卡片，自己挑隊 | 朋友約好要同隊、友誼賽對抗 |
| **隨機分配** | `random` | 報名時自動平衡分配，顯示結果 | 輕鬆揪團、彼此不認識 |
| **主辦分配** | `manual` | 報名時不分隊，由主辦事後處理 | 教練依技能分組、特殊考量 |

## UX 設計

### 一、建立活動時 — 分隊設定

在建立表單中加一個**折疊區塊**（預設收合，不增加既有表單複雜度）：

```
┌─────────────────────────────────┐
│ 👕 分隊模式                  ▸  │  ← 折疊開關
├─────────────────────────────────┤
│                                 │
│  隊伍數量    [ 2 ▾ ]           │  ← 下拉 2/3/4
│                                 │
│  🔴 A 隊    🔵 B 隊            │  ← 色票，點擊換色
│  "紅隊"     "藍隊"             │  ← 可選自訂名稱
│                                 │
│  報名時用戶可以：               │
│  ┌──────┐ ┌──────┐ ┌──────┐   │
│  │ 自選 │ │ 隨機 │ │ 不選 │   │  ← 三選一
│  └──────┘ └──────┘ └──────┘   │
│                                 │
│  ☐ 各隊人數上限均分             │  ← 可選，防止一邊爆滿
│                                 │
└─────────────────────────────────┘
```

**預設值**：2 隊、紅 🔴 + 藍 🔵、隨機模式、均分開啟。主辦開啟開關後幾乎不用改。

**色票選擇**：提供 8 個預設色（對應常見球衣顏色），點擊循環切換：

```
🔴 紅  🔵 藍  🟢 綠  🟡 黃  ⚪ 白  ⚫ 黑  🟠 橙  🟣 紫
```

### 二、用戶報名時

#### 模式 A：自選隊伍 (`self-select`)

在報名按鈕**上方**插入選隊區塊：

```
┌─────────────────────────────────┐
│        選擇你的隊伍 👕          │
│                                 │
│  ┌─────────┐    ┌─────────┐    │
│  │  🔴     │    │  🔵     │    │
│  │  紅隊   │    │  藍隊   │    │
│  │  6/10人 │    │  8/10人 │    │
│  │  ████░░ │    │  ████████│    │
│  └─────────┘    └─────────┘    │
│       ↑ 已選                    │
│                                 │
│  ╔═══════════════════════════╗  │
│  ║     ✨ 立即報名 ✨        ║  │
│  ╚═══════════════════════════╝  │
└─────────────────────────────────┘
```

- 色票卡片要**大且顏色飽和**，一眼就懂
- 顯示目前人數 + 進度條，讓用戶自然看到平衡狀況
- 選中的卡片加粗框 + 微放大動畫
- 開啟均分上限時，滿的隊伍卡片變灰 + 顯示「已滿」
- 未選隊時報名按鈕 disabled

#### 模式 B：隨機分配 (`random`)

報名流程完全不變，報名成功後的 Toast 多一行：

```
┌─────────────────────────────┐
│  ✅ 報名成功！               │
│  你被分配到 🔴 紅隊          │
└─────────────────────────────┘
```

隨機演算法為**平衡分配**：放進目前人數最少的隊，而非純 `Math.random()`。

#### 模式 C：主辦分配 (`manual`)

報名流程完全不變，用戶在活動詳情頁看到：

```
│  👕 分隊狀態：主辦安排中...    │
```

主辦分配完成後才會顯示隊伍結果。

### 三、活動詳情頁 — 分隊展示

不管哪種模式，結果展示統一：

│                                 │

- 用戶自己所在的隊伍排最上方，名字旁加 ✦ 標記

### 四、主辦管理端 — 分隊管理面板

```
┌───────────────────────────────────────┐
│  👕 分隊管理                          │
│                                       │
│  ┌──────────┐   ┌──────────┐         │
│  │ 🔴 紅隊  │   │ 🔵 藍隊  │         │
│  │ 8/10 人  │   │ 7/10 人  │         │
│  ├──────────┤   ├──────────┤         │
│  │ 小明     │   │ 小花     │         │
│  │ 阿華  ⇄ │   │ 阿強     │  ← 點 ⇄ 換隊
│  │ 大毛     │   │ 美美     │         │
│  │ ...      │   │ ...      │         │
│  └──────────┘   └──────────┘         │
│                                       │
│  ⚪ 未分配 (3人)                      │
│  │ 新人 │ 小白 │ 阿龍 │              │
│                                       │
│  ┌─────────┐ ┌─────────┐ ┌────────┐ │
│  │ 🎲 隨機 │ │ ⚖️ 平衡  │ │ 🔄 重置│ │
│  │  全部重分│ │  補齊未分│ │  清空  │ │
│  └─────────┘ └─────────┘ └────────┘ │
└───────────────────────────────────────┘
```

#### 快捷操作：

| 按鈕 | 功能 | 說明 |
|------|------|------|
| 🎲 隨機全部重分 | 忽略現有分配，全部打散重分 | 二次確認防誤觸 |
| ⚖️ 平衡補齊 | 只對「未分配」的人隨機分隊 | 保留已有分配 |
| 🔄 重置 | 全部移到「未分配」 | 重來，二次確認 |

#### 單人操作：

- 點人名旁的 ⇄ 按鈕 → 底部彈出色票卡片選擇目標隊伍
- 操作即時生效，無需額外儲存按鈕

## 邊界情境處理

### 候補遞補時的隊伍分配

| 模式 | 遞補行為 |
|------|----------|
| 自選 | 候補者報名時已選隊 → 遞補後保留原選擇 |
| 隨機 | 遞補時自動分到人數最少的隊 |
| 主辦分配 | 遞補後進入「未分配」，等主辦處理 |

### 同行者（Companion）

- **預設**：同行者跟主報名人同隊
- **自選模式**：可展開個別調整（進階選項，預設收合）
- **原因**：朋友/家人通常想在一起，減少操作步驟

### 取消報名

- 取消者從隊伍中移除
- 不觸發自動重新平衡（避免其他人被換隊造成混亂）
- 主辦可手動用 ⚖️ 平衡補齊

### 容量調整

- 主辦修改 max 人數時，均分上限自動重算（`Math.floor(max / teamCount)`）
- 不自動移動已分配的人，僅影響後續報名

### 多日活動

- 每場獨立分隊（不同天可能不同人來）
- 可選「沿用上次分隊」（未來擴充，初版不做）

## 與既有功能整合

| 功能 | 整合方式 |
|------|----------|
| **簽到簽退表** | 人名前加色點 badge，簽到時一目瞭然哪隊 |
| **LINE 分享卡片** | Flex Message 增加一行「隊伍分佈：🔴 8 vs 🔵 7」 |
| **活動列表卡片** | 不顯示分隊資訊（資訊密度已夠高），進詳情才看 |
| **活動模板** | 模板存分隊設定（隊數、顏色、模式），不存人員分配 |
| **成就系統** | 未來可擴充「連續 N 場不同隊」等趣味成就 |

## 資料模型

### Event 文件擴充

```javascript
event.teamSplit = {
  enabled: true,                              // 是否啟用分隊
  mode: 'self-select' | 'random' | 'manual', // 報名時預設行為
  balanceCap: true,                           // 各隊人數上限均分
  teams: [
    { key: 'A', color: '#EF4444', name: '紅隊' },
    { key: 'B', color: '#3B82F6', name: '藍隊' }
  ]
}
```

### Registration 文件擴充

```javascript
registration.teamKey = 'A' | 'B' | null  // null = 未分配
```

### 設計考量

- 不需要新 Firestore collection，僅在既有文件上加欄位
- `teamSplit` 為 null 或 `enabled: false` 時，整個功能隱藏，零影響
- `registration.teamKey` 寫入時機依模式而定（自選：報名時、隨機：報名時、主辦：管理端操作時）
- 管理端調整隊伍時，直接 update `registration.teamKey`，不需要 transaction（非金融操作，最終一致即可）
