📊 執行摘要 (Executive Summary)

核心發現:ToosterX 在台灣 OSS 圈是孤狼。9 個 agent 跨 8 個維度搜尋,**沒有任何台灣專案同時做到「LIFF + Firebase + 運動 + 報名 + 候補 + 簽到 + 成就 + 賽事」**。最像的對手不是少了 LIFF、就是少了 Firebase、就是少了運動垂直。

3 個層次的分布

  • 同領域(運動報名):台灣有 ~10 個專案,但都是單一運動(羽球居多)、單一功能(只報名 OR 只簽到 OR 只排點),**沒人做整合套件**。
  • 同技術棧(LIFF + Firebase):台灣有 ~15 個專案,但都是教學範例 / 個人小工具 / 業務名片 / 教學機器人,**沒人做大型業務系統**。
  • 同架構模式(Vanilla JS + 無 build + Cloudflare Pages):全網 + 台灣加起來 < 5 個專案,**ToosterX 是這個交叉口最大的一個**。

意外的「市場空白」

  • 2024 亞洲匹克球賽剛在台中辦完、2025 CAPA Open 即將舉行,**台灣完全沒有匹克球專屬 App**(羽球反而有 3 個:羽球GO / PlayOne / 揪波)。
  • 台灣 LINE 開發圈雖然活躍(LINE 台灣有官方 evangelist + GDE + API Expert),但**運動垂直完全沒人開源做**。
  • 台灣 dev 普遍**不愛開源** — `awesome-tw-foss` 全清單只有 ~50 個專案。

調查方法

9 個 AI agent 並行跑,各負責不同維度:LIFF 深度 / 運動 zh-TW / Firebase Vanilla / QR 簽到 / 成就 EXP / Awesome+閉源 / 國際同類 / Cloudflare 架構 / 缺口補搜。每個 agent 各自做 1500-2500 字深度報告,最後合併去重。

🚨 關鍵警告 (已重新驗證,2026-04-27 更新)

✅ 已澄清 - LIFF QR 掃描 API「過時」是誤報

原本警告 liff.scanCodeV2() 已被 LINE deprecated。經第二輪深度查證(LINE 官方 LIFF API 參考、release notes、deprecation 公告頁)確認此說法錯誤。

真相:被 deprecated 的是更舊的 liff.scanCode()(V1)。scanCodeV2()(V2)是 LINE 為了修復 V1 在 iOS 失效而推出的「修復方案」,**截至 LIFF v2.28.0(2026-03-24)仍是現役推薦方法**,LINE 官方 demo `line/liff-playground` 自己也用它。

ToosterX 額外確認:即使該警告屬實,ToosterX 也根本沒用 LIFF 內建掃描 — js/modules/scan/scan-camera.js 使用 html5-qrcode 2.3.8(跨瀏覽器標準),零相關。

結論:不需要任何修改。原警告來自 AI agent 幻覺,**警告涉及 production code 修改前必須驗證第一手來源**。

⚠️ 嚴重(實測比想像更高) - 多數台灣 repo 沒授權,商業 copy 是刑事責任

原本估計 80%,**實測 21 個調查到的台灣 repo,95.2%(20/21)沒 LICENSE**。GitHub 官方 + 台灣著作權法雙重確認:沒 LICENSE = 保留所有權。

台灣著作權法量級:逐字 copy 是 重製權侵害(著作權法第 22 條)。商業重製最高 5 年徒刑 + 500 萬罰金(著作權法第 91 條)。第 37(1)條:「約定不明之部分,推定為**未授權**」。

但好消息:著作權法第 10-1 條明文「思想、概念、原理、操作方法、系統」**不受著作權保護**。所以「**讀別人 repo 學 idea → 自己重寫**」完全合法。ToosterX 既有開發模式不需改變。

唯一例外要小心:`CXPhoenix/roll-call-remote-class` 是調查到唯一有 LICENSE 的台灣 repo,但是 GPL-3.0(copyleft 強傳染) — 一沾到你 ToosterX 整個都要被迫開源 GPL。**商業化路線絕對不要 copy 任何 GPL 程式碼**。

📌 關於 ToosterX 自己的 LICENSE 狀態

ToosterX 目前**也沒寫 LICENSE**(同 95% 台灣 repo)。**用戶決策:走商業化路線,維持「source-available, all rights reserved」狀態**(見 README.md)。這代表:

  • ✅ 別人不能合法 copy / fork / 商用你的 code(商業護城河)
  • ✅ 你保留所有商業選項(將來可改 license / 賣授權 / 拿投資)
  • ⚠️ 不能上 awesome-tw-foss 等開源清單(因為法律上不算開源)
  • ⚠️ 不會吸引外部 contributor(這對商業化反而是好事)

3. 觀察 3 個台灣即時對手

每月看一次:kent0925/Activity(2026-04-27 push)、d336643/NTU_badminton_system(NTU 9★)、subcer/badminton_lineup(Vanilla+Firebase 同棧)。看他們動什麼,提早決策要不要跟。

4. 寫一篇 Medium 長文(技術角度,不開源)

「用 Firebase + LIFF 從零打造運動社群」 — **同領域沒人寫過產品級實作**,搜尋紅利能吃 1-2 年。寫**架構決策 + 踩雷分享**而不秀整套 code,既能拿 dev 圈名聲又不洩漏商業 know-how。比 iThome 鐵人賽好(後者要 30 篇,長文 1 篇就夠)。

🔴 「不該做」清單

1. 不要上 awesome-tw-foss / AwesomeChatbots(與商業化衝突)

原本第 1 項建議已撤銷。**這些清單明確要求 OSS license**,你維持「source-available, all rights reserved」狀態無法上(法律上不算開源),硬要上會被視為「假開源」傷品牌。

2. 不要 copy 任何沒 LICENSE 的台灣 repo 程式碼

實測 95.2% 沒 license = 保留所有權,**商業 copy 觸著作權法第 91 條,最高 5 年徒刑 + 500 萬罰金**。可以「讀別人 repo 學 idea → 自己重寫」(著作權法第 10-1 條保護「概念不保護表達」),但不能直接複製。

3. 絕對不要 copy 那個唯一有 LICENSE 的 GPL-3.0 repo

`CXPhoenix/roll-call-remote-class` 是調查到唯一有授權的台灣 repo,但 GPL-3.0 是**強傳染 copyleft**。一沾上整個 ToosterX 都要被迫開源 GPL,商業化路線等於自殺。

4. 不要立刻轉框架(React / Vue / Next.js)

你的「Vanilla + 無 build」是**少數人才能做到的紀律**,商業上有差異化價值。除非你要從個人開發轉為 4-5 人團隊,否則保持現狀。
