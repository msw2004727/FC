# 00 Overview

## 目標
將目前偏單頁展示型的 `賽事（tournaments）` 功能，重構成可承載正式賽事流程的子系統。第一階段只交付 `友誼賽（friendly）`，但資料模型、權限模型、UI 掛點與通知 key 一開始就要能延伸到 `盃賽（cup）` 與 `聯賽（league）`。

## 第一階段成功標準
- 符合資格的使用者可以建立友誼賽
- 主辦方與委託人可以在賽事詳情頁審核俱樂部申請
- 已核准俱樂部的隊員可以加入或退出 roster
- 多俱樂部成員可在加入 roster 前選擇代表隊伍
- 第三方不可見候審中的俱樂部
- 站內信模板可支援建賽、申請、核准、拒絕四類核心通知

## 核心產品決策
- 賽事主辦單位是 `俱樂部（hostTeam）`，不是個人
- UI 顯示主辦方時固定為 `俱樂部名（用戶名）`
- 一般使用者建立賽事的資格是：
  - 擁有俱樂部
  - 且在俱樂部中是 `領隊（captain）` 或 `經理（leader / leaderUids）`
- `管理員（admin）`、`總管（super_admin）` 可跳過上述建賽資格限制，但仍必須指定主辦俱樂部
- `教練（coach）` 不納入第一階段的建賽或代表俱樂部報名資格
- 主辦俱樂部在建立友誼賽後自動成為第一支已核准隊伍
- 其他俱樂部必須先由 `captain / leader` 送出俱樂部申請，經主辦核准後，隊員才可加入該隊 roster
- 候審中的俱樂部不佔 4 隊名額，但主辦在核准瞬間必須檢查名額

## 第一階段非目標
- 不交付完整 `cup` bracket
- 不交付完整 `league` standings / schedule
- 不處理賽事結束後的戰績、積分、淘汰結果寫回
- 不處理裁判、場務、賽程衝突演算法

## 名詞
- `建立者（creator）`
  - 實際按下建立賽事的人
- `主辦俱樂部（hostTeam）`
  - 真正擁有該賽事的俱樂部單位
- `主辦顯示（organizerDisplay）`
  - UI 字串，格式固定為 `俱樂部名（用戶名）`
- `委託人（delegates）`
  - 擁有單一賽事範圍管理權的受委託使用者
- `俱樂部申請（team application）`
  - 某支俱樂部申請參賽、等待主辦審核的記錄
- `參賽隊伍（team entry）`
  - 已經被核准、正式列入賽事的隊伍
- `隊員名單（member roster）`
  - 某支參賽隊伍實際要出賽的成員列表

## 產品限制
- `友誼賽（friendly）` 第一階段上限固定 4 隊
- 仍需保留未來多隊延伸空間，不能在資料或 UI 中把 4 寫死成唯一結構
- 延續現有 `站內信模板（notifTemplates）` 機制，由 `站內信管理（page-admin-messages）` 編輯內容
