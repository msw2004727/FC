## 七、風險與對策

### 風險 1：A1（完全不等動畫）視覺尷尬

**情境**：按鈕從「報名中」變「取消報名」、但翻牌動畫還在跑、可能短暫並存。

**對策**：
- 改 CSS 讓動畫不影響按鈕子元素的 visibility
- 或用 A2（等 600ms 折衷）— 動畫跑一半時 UI 更新

### 風險 2：B（minInstances=1）月成本

**情境**：每月固定 ~$5-10 美金、即使無人報名也付。

**對策**：
- 接受成本（換體感）
- 或設定**僅活動高峰時段**保溫（透過 Cloud Scheduler 動態調整、複雜）
- 或只給 `registerForEvent` 設、`cancelRegistration` 用率低可不設（折半成本）

### 風險 3：C（locked function 改動）

**屬 CLAUDE.md §報名系統保護規則 鎖定範圍**：
- 必須跑 `npm run test:unit`（550+ 報名相關測試）
- 必須在 console 跑 `docs/registration-integrity-check.js`
- 必須加複合索引、更新 `firestore.indexes.json`
- 改錯可能造成超收/重複報名

**對策**：先做 A + B、如果體感已達「秒成功」就不碰 C。

### 風險 4：minInstances 在低流量時段資源浪費

**情境**：凌晨 3 點沒人報名、但 CF instance 一直開著付費。

**對策**：可接受（每月 $5-10 比客戶抱怨慢便宜）。
