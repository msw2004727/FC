### Codex 審查補充備註(請 Claude 實作前嚴格核實)

> 以下 5 點不是推翻本計劃書的問題分類,而是 Codex 重新對照同一 worktree 的前端、`ApiService`、`FirebaseService` 與 `firestore.rules` 後,發現「照原方案直接做」可能仍會失敗或產生前後端不一致。Claude 實作時請先逐點核實屬實與否,再調整修補方案。

1. **P1/root + host entry 原子建立方案不可只做 client batch**
   - 本計劃書 P1「建立 root + host entry 原子化」提議用 batch 同時建立 `tournaments/{id}` 與 `tournaments/{id}/entries/{hostTeamId}`。
   - 但目前 `firestore.rules` 的 `entries create` 仍要求 `tournamentExists() && canManageTournamentScope()`；若父文件也是同一 batch 才新建,現行 `exists/get` 型判斷不應視為已存在。
   - 修正方向:優先改成 Cloud Function/Admin SDK 原子建立；若堅持 client batch,必須同步把 rules 改成可安全驗證同批寫入的 `getAfter()` 設計,並補完整 host entry 欄位、`hostTeamId`、`creatorUid`、`teamId` 一致性驗證。不能只新增 client batch。

2. **P1/approve transaction 範例不是安全可落地版本**
   - 本計劃書範例在 transaction 內使用 `tx.get(tournamentRef.collection('entries'))` 讀整個 entries collection,這不應作為 Web client transaction 的落地基準。
   - `newRegisteredTeams = [...entriesSnap.docs.map(d => d.id), application.teamId]` 也缺少 host/approved 過濾、去重與同隊已存在時的冪等處理。
   - 修正方向:優先移到 Cloud Function/Admin SDK 做審核原子交易；或改成 root 文件維護 `approvedTeamCount` / `registeredTeams` summary,transaction 只讀寫固定 document refs,避免在 client transaction 中掃 collection。

3. **P1/applications rules 範例語法不可直接使用**
   - 本計劃書 rules 範例的 `data.keys().hasOnly([...prev.keys(), ...])` 使用了 JavaScript spread 語法,Firestore Rules 不支援,會導致 rules 編譯失敗。
   - 修正方向:update 應使用 `request.resource.data.diff(resource.data).affectedKeys().hasOnly([...])`；同時補上 `id == applicationId`、`teamId/requestedByUid` 不可變、`status` 狀態轉移、`createdAt/updatedAt/reviewedAt` 型別或 `request.time` 驗證。

4. **P2/application list 修補少了 ApiService wrapper 與使用者隊伍脈絡**
   - 本計劃書範例呼叫 `ApiService.getTournamentApplication(...)`,但目前只看到 `FirebaseService.list/create/updateTournamentApplication` 與 `ApiService.list/create/updateTournamentApplication`,尚未有 `ApiService.getTournamentApplication` wrapper。
   - 範例也使用 `currentUser` 但片段沒有先定義；且只查 `_getFriendlyResponsibleTeams(currentUser)`,可能漏掉 rules 已允許 `isCurrentUserInTeam(resource.data.teamId)` 讀取的「一般隊員看自己隊伍申請」情境。
   - 修正方向:同步補 `FirebaseService.getTournamentApplication` 與 `ApiService.getTournamentApplication`；並明確決定產品規格:若一般隊員應看得到自己隊伍 pending,查詢 doc id 時要納入 current user 所屬 teamIds,不只 captain/leader responsible teams。

5. **P2/create guard 可能把 admin 擋在前端**
   - 本計劃書建議把建立賽事 guard 改成 `manage_all OR _canCreateFriendlyTournament(createUser)`。
   - 但目前 `_canCreateFriendlyTournament` 只接受 `admin.tournaments.manage_all` 或隊長/領隊；同時預設 admin 權限不一定包含 `admin.tournaments.manage_all`,而 rules 的 root create 又允許 `isAdmin()`。
   - 修正方向:前端 create guard 必須與 rules 對齊。可選做法是讓 admin role 或等價 admin 判斷也通過 `_canCreateFriendlyTournament`,或在 handler 額外允許明確的 admin 判斷；不要造成 rules 允許但前端擋住 admin 的不一致。

### Codex 第三輪審查補充備註(2026-04-28,請 Claude 實作前嚴格核實)

> v2 已吸收主要方向,但 Cloud Function 草案與前端權限對齊仍有可落地風險。以下 5 點請視為實作硬性條件,不是選配註記。

1. **P1/Cloud Function 草案必須改用本專案 Firebase Functions v2 寫法**
   - 本專案 `functions/index.js` 已使用 `const { onCall, onRequest, HttpsError } = require("firebase-functions/v2/https")`、`const { getFirestore, FieldValue } = require("firebase-admin/firestore")`、`const db = getFirestore()`。
   - P1-⑤/P1-⑥ 草案若出現 `functions.region(...).https.onCall`、`functions.https.HttpsError`、`admin.firestore()`、`admin.firestore.FieldValue.serverTimestamp()` 都是 v1/不存在物件寫法,不可照貼。
   - 正確方向:`exports.xxx = onCall({ region: "asia-east1", timeoutSeconds: 30, memory: "512MiB" }, async (request) => { ... })`;使用 `request.auth` / `request.data`、`new HttpsError(...)`、`db.collection(...)`、`FieldValue.serverTimestamp()`。

2. **P1/create callable 不得信任 client payload 或 client 指定 id**
   - 前端可傳 `tournament` 草稿,但後端必須白名單重建可寫欄位,不可 `batch.set(tournamentRef, { ...tournament })`。
   - 賽事 doc id 必須由後端產生,建議 `const tournamentRef = db.collection("tournaments").doc()` 後用 `batch.create(tournamentRef, safeTournamentPayload)`。不可接受 `tournament.id` 覆寫既有 doc。
   - `hostEntry` 也不可 optional 或完全由 client 傳入。Cloud Function 必須依已驗證的 `hostTeamId` / team doc / caller user 在後端產生 host entry,並與 root 同一 batch 原子 `create`。

3. **P1/Function 權限 helper 必須對齊專案既有身份解析與 rules**
   - 不要直接假設 `users/${uid}` 一定存在。functions 端既有模式是 `findUserDocByUidOrLineUserId(uid)`、`getCallerRoleWithFallback(request)`。
   - team officer 判斷不可只看 `captainUid` + `leaderUids`。需對齊 `firestore.rules` 的 `isCurrentUserTeamCaptainOrLeader(teamId)`: `captainUid`、`creatorUid`、`ownerUid`、`leaderUid`、`leaderUids` 都要納入。
   - 建議在 `functions/index.js` 抽 server-side helper,例如 `isTournamentTeamOfficerForData(team, uid)` 與 `isRoleAdminOrAbove(role)`。

4. **P0/admin role 對齊不能只修 create**
   - 若只讓 `_canCreateFriendlyTournament()` 通過 admin role,admin 可能「能建立」但仍不能進入管理列表、讀 application list、編輯、審核、結束。
   - 前端需抽共用 helper,例如 `_isTournamentGlobalAdmin(user)` / `_canGloballyManageTournaments()`。
   - `handleCreateTournament`、`_canCreateFriendlyTournament`、`_canManageTournamentRecord`、`renderTournamentManage` 的 `isAdmin`、`showEditTournament/handleSaveEditTournament`、`reviewFriendlyTournamentApplication`、`handleEndTournament` 都要使用同一個 global-admin 判斷,避免 create/manage/review/end 前後不一致。

5. **P1/前端呼叫 atomic create 時必須補 ApiService wrapper**
   - 若 `handleCreateTournament` 改呼叫 `ApiService.createFriendlyTournamentAtomic(createData, hostEntry)`,必須同步在 `api-service.js` 補同名 wrapper。
   - 寫法對齊既有 tournament API:先 normalize/保留本地需要的欄位,再呼叫 `FirebaseService.createFriendlyTournamentAtomic(...)`。不可只在 `firebase-crud.js` 新增方法。

### Codex 第四輪審查補充備註(2026-04-28,請 Claude 實作前嚴格核實)

> v3-impl 已吸收主要方向,但仍有 4 個「照草案實作會出錯」的落地風險。以下列為硬性修正,不是產品偏好。

1. **P1/create callable 產生 id 必須維持 `ct_...` 契約**
   - `CLAUDE.md` 已明定賽事唯一識別使用 `ct_` 前綴,且 [app.js](app.js) 的 `applyTournament` deep action 會拒絕非 `ct_` id。
   - Cloud Function 不可用 `db.collection('tournaments').doc()` 的隨機 auto id 作為 `tournament.id`。
   - 正確方向:後端自行產生 `ct_${Date.now()}_${random}` 格式 id,再用 `db.collection('tournaments').doc(tournamentId)` + `batch.create()`。這仍能避免覆寫既有文件。

2. **P1/`findUserDocByUidOrLineUserId()` 回傳結構不可讀錯**
   - functions 內此 helper 回傳 `{ docId, data }`。
   - `callerUserDoc?.displayName` / `reviewerUserDoc?.displayName` 會取不到名稱。
   - 正確方向:讀 `callerUserDoc?.data?.displayName || callerUserDoc?.data?.name` 與 `reviewerUserDoc?.data?.displayName || reviewerUserDoc?.data?.name`,或沿用既有 `resolveAuditActorName()` 類似邏輯。

3. **P1/create callable 需由後端 derive/覆寫關鍵狀態欄位**
   - 白名單不可包含或直接信任會影響流程與權限判斷的欄位:`status`、`ended`、`mode`、`typeCode`、`type`、`friendlyConfig`、`hostTeamId`、`hostTeamName`、`hostTeamImage`。
   - 後端應使用 trim 後的 `hostTeamId`、team doc、validated teamLimit 與 regStart/regEnd 重建這些欄位。
   - 最低要求:`mode/typeCode/type = friendly`、`ended = false`、`teamLimit` 夾在 2-4、`friendlyConfig` 由後端建立、`status` 由時間或限定白名單計算。

4. **P1/review 前端 callsite 不可 cache-only**
   - `ApiService.getTournament(tournamentId)` 只讀快取,若詳情狀態尚未注入或被分頁 onSnapshot 洗掉,合法 creator/delegate/host manager 可能被前端誤擋。
   - 正確方向:保留既有 `_loadFriendlyTournamentDetailState(tournamentId)` 後再做權限與 pending application 檢查,或至少 fallback `await ApiService.getTournamentAsync(tournamentId)`。
   - 同時保留 reject confirm、pending application 檢查與 action 白名單,不要只把按鈕直接接到 callable。

### Codex 第五輪審查補充備註(2026-04-28,請 Claude 實作前嚴格核實)

> v3-codex4 仍有 3 個前後端一致性風險。以下不是新增功能,而是避免照計劃實作後仍留下可繞過路徑或權限誤判。

1. **P1/applications client update 不可繞過 callable 原子審核**
   - P1-⑥ 已把 approve/reject 改成 Cloud Function,目標是 application、entry、registeredTeams summary 同步原子更新。
   - 因此 P1-⑦ 的 Firestore rules 不應再允許 Web client 直接把 `applications/{id}.status` 改成 `approved/rejected`。
   - 正確方向:client rules 對 applications update 先採 `allow update: if false`;若未來有 applicant cancel,另開只允許 applicant 改 `status: 'cancelled'` 的白名單,不得允許 review 欄位。

2. **P2/前端權限 helper 必須先定義 `manage_all` 產品語意**
   - 目前 rules 的 `isAdmin()` 是 role-based(`admin/super_admin`),不是 `hasPermission('admin.tournaments.manage_all')`。
   - 若前端 `_isTournamentGlobalAdmin()` 把非 admin 但有 `manage_all` 的使用者放行,callable/rules 仍可能擋掉,造成前端可進、後端失敗。
   - 本計劃預設採「role-based global admin」對齊現行 rules；若產品決定 `manage_all` 是可下放的全域賽事管理權,必須同步補 `firestore.rules` 與 callable server helper,不可只改前端。
   - `_getFriendlyResponsibleTeams()` 也不可只看 captain/leader,必須對齊 rules/callable 的 5 欄位:`captainUid`、`creatorUid`、`ownerUid`、`leaderUid`、`leaderUids`。

3. **P2/create callable 必須做後端日期驗證**
   - 前端既有 `regStart < regEnd` 檢查不能被 callable 信任。
   - Cloud Function 需解析 `regStart/regEnd`,要求兩者皆為有效日期且 `regStart < regEnd`,再用驗證後值 derive `status`。
   - 惡意 caller 帶空值、無效日期或 `regStart >= regEnd` 時,callable 必須回 `invalid-argument`,不可建立賽事 root/entry。

    const regStart = String(tournamentInput.regStart || '').trim();
    const regEnd = String(tournamentInput.regEnd || '').trim();
    const regStartDate = new Date(regStart);
    const regEndDate = new Date(regEnd);
    if (!regStart || !regEnd
        || Number.isNaN(regStartDate.getTime())
        || Number.isNaN(regEndDate.getTime())
        || regStartDate >= regEndDate) {
      throw new HttpsError('invalid-argument', '報名時間不合法');
    }

    // 2) caller 身份解析(對齊 functions 既有 helper)
    const callerUserDoc = await findUserDocByUidOrLineUserId(callerUid);
    const callerRole = await getCallerRoleWithFallback(request);
    const callerName = String(callerUserDoc?.data?.displayName || callerUserDoc?.data?.name || '').trim();
    const isAdmin = isRoleAdminOrAbove(callerRole);

    // 3) host team 存在驗證 + officer 檢查(5 欄位完整)
    const teamSnap = await db.doc(`teams/${hostTeamId}`).get();
    if (!teamSnap.exists) {
      throw new HttpsError('not-found', '主辦俱樂部不存在');
    }
    const teamData = teamSnap.data();
    const isHostOfficer = isTournamentTeamOfficerForData(teamData, callerUid);
    if (!isAdmin && !isHostOfficer) {
      throw new HttpsError('permission-denied', '只有主辦俱樂部的隊長 / 領隊 / 創辦人可建立賽事');
    }

    // 4) 白名單重建 payload(不接受 client 偽造 creatorUid / id / createdAt 等)
    const safePayload = {};
    for (const key of TOURNAMENT_WRITE_ALLOWED_FIELDS) {
      if (tournamentInput[key] !== undefined) safePayload[key] = tournamentInput[key];
    }
    const teamLimit = sanitizeTournamentTeamLimit(
      tournamentInput?.friendlyConfig?.teamLimit
        ?? tournamentInput.teamLimit
        ?? tournamentInput.maxTeams
        ?? tournamentInput.teams,
      4
    );

    // 5) 後端產生 ct_ 格式 doc id(不信任 client tournament.id,且維持專案 ID 契約)
    const tournamentId = generateTournamentId();
    const tournamentRef = db.collection('tournaments').doc(tournamentId);
    const entryRef = tournamentRef.collection('entries').doc(hostTeamId);

    // 6) 後端產生 host entry(不信任 client hostEntry 整包)
    const hostEntry = buildServerHostEntry({
      teamId: hostTeamId,
      teamData,
      callerUid,
      callerName,
    });

    // 7) Admin SDK batch.create(若 doc 已存在會拋錯,確保不覆寫)
    const batch = db.batch();
    batch.create(tournamentRef, {
      ...safePayload,
      id: tournamentId,
      mode: 'friendly',
      typeCode: 'friendly',
      type: '友誼賽',
      regStart,
      regEnd,
      teams: teamLimit,
      maxTeams: teamLimit,
      teamLimit,
      hostTeamId,
      creatorUid: callerUid,        // 強制覆寫,防偽造
      creatorName: callerName,
      hostTeamName: String(teamData?.name || '').trim(),
      hostTeamImage: String(teamData?.image || '').trim(),
      friendlyConfig: {
        teamLimit,
        allowMemberSelfJoin: true,
        pendingVisibleToThirdParty: false,
      },
      registeredTeams: [hostTeamId], // 初始 summary
      status: resolveTournamentStatusByTime(regStart, regEnd),
      ended: false,
      schemaVersion: 2,
      dataModel: 'tournament_v2',
      createdAt: FieldValue.serverTimestamp(),
      updatedAt: FieldValue.serverTimestamp(),
    });
    batch.create(entryRef, hostEntry);

    await batch.commit();

    return { tournamentId };
  }
);

```javascript
// functions/index.js
exports.reviewFriendlyTournamentApplication = onCall(
  { region: 'asia-east1', timeoutSeconds: 30, memory: '512MiB' },
  async (request) => {
    if (!request.auth?.uid) {
      throw new HttpsError('unauthenticated', '請先登入');
    }
    const reviewerUid = request.auth.uid;
    const { tournamentId, applicationId, action } = request.data || {};

    // 1) action 白名單(對應 P2-⑨,後端強制驗證)
    const normalizedAction = String(action || '').trim().toLowerCase();
    if (!['approve', 'reject'].includes(normalizedAction)) {
      throw new HttpsError('invalid-argument', '未知的審核操作');
    }
    if (!tournamentId || !applicationId) {
      throw new HttpsError('invalid-argument', '缺少 tournamentId 或 applicationId');
    }

    // 2) reviewer 身份解析(對齊既有 helper,防 LINE uid / legacy uid 漏判)
    const reviewerUserDoc = await findUserDocByUidOrLineUserId(reviewerUid);
    const reviewerRole = await getCallerRoleWithFallback(request);
    const reviewerName = String(reviewerUserDoc?.data?.displayName || reviewerUserDoc?.data?.name || '').trim();
    const isAdmin = isRoleAdminOrAbove(reviewerRole);

    const tournamentRef = db.doc(`tournaments/${tournamentId}`);
    const applicationRef = tournamentRef.collection('applications').doc(applicationId);

    return db.runTransaction(async (tx) => {
      const [tournamentSnap, appSnap] = await Promise.all([
        tx.get(tournamentRef),
        tx.get(applicationRef),
      ]);
      if (!tournamentSnap.exists) throw new HttpsError('not-found', '賽事不存在');
      if (!appSnap.exists) throw new HttpsError('not-found', '申請不存在');

      const tournament = tournamentSnap.data();
      const application = appSnap.data();

      // 3) 權限:admin / creator / delegate / host team officer(5 欄位完整)
      const isCreator = tournament.creatorUid === reviewerUid;
      const isDelegate = Array.isArray(tournament.delegateUids) && tournament.delegateUids.includes(reviewerUid);
      let isHostOfficer = false;
      if (tournament.hostTeamId) {
        const teamSnap = await tx.get(db.doc(`teams/${tournament.hostTeamId}`));
        if (teamSnap.exists) {
          isHostOfficer = isTournamentTeamOfficerForData(teamSnap.data(), reviewerUid);
        }
      }
      if (!isAdmin && !isCreator && !isDelegate && !isHostOfficer) {
        throw new HttpsError('permission-denied', '無權審核此申請');
      }

      // 4) 冪等:若 application 已非 pending,直接回現況不重複寫
      if (application.status !== 'pending') {
        return { status: application.status, alreadyReviewed: true };
      }

      // 5) approve 路徑:讀 entries 做名額檢查 + 寫 entry + 更新 summary
      if (normalizedAction === 'approve') {
        // Admin SDK transaction 支援 collection query(client SDK 不支援)
        const entriesSnap = await tx.get(tournamentRef.collection('entries'));
        const approvedTeamIds = entriesSnap.docs
          .filter(d => {
            const s = d.data()?.entryStatus;
            return s === 'host' || s === 'approved';
          })
          .map(d => String(d.data()?.teamId || '').trim())
          .filter(Boolean);
        const teamLimit = Number(
          tournament.friendlyConfig?.teamLimit
          || tournament.maxTeams
          || tournament.teamLimit
          || 4
        );

        // 冪等:該隊已 approved → 只更新 application,不重複建 entry
        if (!approvedTeamIds.includes(application.teamId)) {
          if (approvedTeamIds.length >= teamLimit) {
            throw new HttpsError('failed-precondition', '隊伍名額已滿');
          }

          tx.set(
            tournamentRef.collection('entries').doc(application.teamId),
            {
              teamId: application.teamId,
              teamName: String(application.teamName || '').trim(),
              teamImage: String(application.teamImage || '').trim(),
              entryStatus: 'approved',
              approvedAt: FieldValue.serverTimestamp(),
              approvedByUid: reviewerUid,
              approvedByName: reviewerName,
              createdAt: FieldValue.serverTimestamp(),
              updatedAt: FieldValue.serverTimestamp(),
            },
            { merge: true }
          );

          // summary:過濾 host/approved + 去重 + append
          const newRegisteredTeams = Array.from(new Set([...approvedTeamIds, application.teamId]));
          tx.update(tournamentRef, {
            registeredTeams: newRegisteredTeams,
            updatedAt: FieldValue.serverTimestamp(),
          });
        }
      }

      // 6) update application(approve / reject 都走這條)
      tx.update(applicationRef, {
        status: normalizedAction === 'approve' ? 'approved' : 'rejected',
        reviewedAt: FieldValue.serverTimestamp(),
        reviewedByUid: reviewerUid,
        reviewedByName: reviewerName,
        updatedAt: FieldValue.serverTimestamp(),
      });

      return { status: normalizedAction === 'approve' ? 'approved' : 'rejected' };
    });
  }
);
```
