有效授權如何組成

這四個來源用途不同，不能只看角色名稱或單一權限陣列判斷使用者能做什麼。

角色rolePermissions
後台入口與管理操作的角色層權限。前端與 Cloud Functions 在文件不存在或缺少 permissions 欄位時使用內建預設；Firestore Rules 的 hasPerm() 則要求文件存在且包含權限陣列，缺檔時直接拒絕。一旦已有有效儲存文件，空陣列仍代表空權限。
一般 user 固定回傳空陣列缺檔可能 UI 允許／Rules 拒絕
個別userPermissionGrants
由 super_admin 對特定 UID 授予允許清單中的權限，不改變角色，也不改寫該角色的預設。一般 user 亦可被個別授權，例如活動黑名單管理。
前端：與角色權限合併後端：hasUserPermissionGrant()
活動roleActivityCapabilities
只控制一般 user 的主辦人範圍活動能力，例如建立基本活動、編輯自己活動、現場操作與設定委託人；它不是後台角色權限。
7 項預設開啟進階功能預設關閉
有效effective permissions
前端以目前角色權限加上個別 UID 授權判斷；Firestore Rules 的 hasEffectivePerm() 同樣合併 hasPerm() 與個別授權。
前端：getCurrentUserEffectivePermissions()後端：hasEffectivePerm()
固有INHERENT_ROLE_PERMISSIONS
coach、captain、venue_owner 目前沒有不可剝奪權限。super_admin 的安全保護為 admin.repair.event_blocklist 與 admin.seo.entry。
範圍入口不等於動作
activity.manage.entry 的目標契約是入口可見性，新路徑不得把它當泛用動作授權；但現行前端與 Rules 仍在多個 owner／delegate scoped helper 內接受 entry、coach+ 或個別授權 fallback。delegate 只界定活動範圍，各動作仍須逐項判斷；目前可依角色、權限或 capability 編輯、取消、重開／重新上架或操作現場，但不因 delegate 身分取得刪除權。

內建角色與目前預設

以下是文件尚未儲存自訂值時的內建基準；實際執行仍須讀取線上設定、個別授權及動作範圍。

一般用戶

Lv.0 · user
7 活動能力預設開啟1 項預設關閉

角色層與活動能力

  • rolePermissions/user 固定為空，不提供角色層後台權限
  • 建立基本活動與外部活動連結
  • 管理、編輯基本資料及取消自己主辦的活動
  • 自己主辦或受委託活動的現場操作；主辦人可設定委託人
  • 收費、私密、隊伍限定、分隊、早鳥與 GPS 等進階功能預設關閉
super_admin 可另外授予特定 UID 允許清單中的權限；這不會改變 user 角色或開啟整個角色層。

一般 user 活動能力

能力由 roleActivityCapabilities/user 控制。表中「開」是文件不存在時的內建預設，線上儲存值可調整。

能力碼預設實際範圍
user.activity.basic_create開建立不含進階功能的基本活動
user.activity.external_create開建立自己主辦的外部活動連結
user.activity.own_manage_entry開顯示自己活動管理入口；不授予任何動作本身
user.activity.own_edit_basic開編輯自己主辦或受委託活動的基本欄位
user.activity.own_cancel開取消自己主辦或受委託的活動
user.activity.site_operate開owner-scope 現場操作；明列 delegate 目前可無此 capability 進入現場操作，但從正取名單移除參與者另判
user.activity.delegate_assign開只有主辦人可設定委託人；delegate 身分本身不授予刪除或設定其他 delegate
user.activity.addons_use關收費、隊伍／性別限定、私密、分隊、社群連結、早鳥與 GPS 等進階功能

管理與驗證原則

權限判斷必須同時核對資料來源、作用範圍與前後端守衛。

預設依授權層分開判斷
角色權限缺檔時，前端與 Functions 使用內建預設，但 Rules 的 hasPerm() fail-closed；一般 user 活動能力缺檔時，前端、Functions 與 Rules 才都有對應預設。線上文件已存在時必須尊重儲存值，空陣列不得補回預設。
user三套控制分離
一般 user 的 rolePermissions 維持空白，但仍可透過 roleActivityCapabilities 管理主辦人範圍功能，或透過 userPermissionGrants 取得特定例外權限。
同步前後端都要授權
前端隱藏按鈕不能取代 Firestore Rules 或 Cloud Functions 驗證；後端允許也不能省略前端可見性與錯誤處理。兩端須使用相同權限碼與作用範圍。
動作使用專用 predicate
建立、編輯、刪除、重開、現場操作、黑名單與高敏感管理必須逐動作判斷。活動入口權限不得作為所有動作的通行證。
個別授權不改角色
個別 UID 授權只增加允許清單中的有效權限，不改使用者 role，也不擴張其他同角色使用者。寫入與刪除僅限 super_admin，並保留稽核資料。
委託delegate 逐動作判斷
一般 user delegate 有 own_edit_basic／own_cancel 時可編輯基本資料／取消；明列 delegate 現行可在沒有 site_operate 時進現場操作，但從正取名單移除參與者仍另判。staff delegate 受現有 entry／coach+ fallback 可編輯、取消、結束與重開／重新上架；delegate 身分本身不授予刪除。
現況活動守衛存在三端 drift
前端與 Rules 的入口守衛仍接受角色 activity.manage.entry、個別 UID grant 與 coach+ fallback，Rules 多數活動更新仍另查 owner／delegate scope。Functions 的 canOperateEventSiteForAccess() 則會先讓 coach+、activity.manage.entry 或 event.edit_all 直接通過而不查該 scope，並被報名取消 Callable 的管理移除／容量調整使用；因此可能形成跨活動名單操作。這是待修的高風險授權差異，不是目標權限設計。
查證以實碼與線上設定為準
權限目錄與預設看 js/config.js；前端有效權限看 js/api-service.js；Firestore 邊界看 firestore.rules；Callable 邊界另查 functions/index.js。線上實際值須再查 rolePermissions、roleActivityCapabilities 與 userPermissionGrants。

一般使用與擁有者範圍功能

下列功能不依賴角色層後台權限，但仍受登入狀態、帳號限制、資料擁有者、活動狀態、可見性及 Firestore Rules 約束。

功能主要邊界說明
瀏覽公開活動與俱樂部公開讀取公開資料可供訪客瀏覽；私密或限定內容另依可見性判斷
報名、取消與候補登入者本人＋活動狀態只能以自己的身分操作，候補遞補由既有流程處理
申請加入或退出俱樂部登入者本人＋成員規則俱樂部幹部、領隊轉讓與冷卻限制另行判斷
管理個人資料與通知設定資料擁有者只可更新 Firestore Rules 允許的自有欄位
查看站內信、出席與成就登入／擁有者範圍依各 collection 的讀取規則與功能狀態提供
